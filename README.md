# Chapter One POS

This repository contains the full end-to-end code for the Chapter One POS application. It relies entirely on natively managed GitHub Releases to deliver production auto-updates smoothly.

## Project Structure
```text
/app               # Shared bootstrap/init scripts bridging Backend & Electron
/backend           # Node.js backend logic, models & localized Express DB API
/config            # Winston configuration and app settings
/electron          # Main process, IPC channels, and Window orchestration
/frontend          # Vite + React interface
/migrations        # `.sql` files executed sequentially by migrationRunner
/scripts           # Developer utility scripts (bumping versions, checkings)
/updater           # Electron-updater GitHub logic and Migration runner
```

## Local Development Setup

1. **Install Dependencies:**
   Ensure Node.js v20+ is installed.
   ```bash
   npm install
   ```

2. **Environment Variables:**
   Copy the `env.example` to `.env` (or create one):
   ```bash
   DB_USER=postgres
   DB_HOST=localhost
   DB_NAME=chapterone_pos
   DB_PASSWORD=your_password
   DB_PORT=5432
   ELECTRON_IS_DEV=true
   ```

3. **Start the Application in Development:**
   ```bash
   npm start
   ```

## GitHub Auto-Update System

This application completely eliminates expensive remote artifact servers. The NSIS installer generated by `electron-builder` natively queries your GitHub API to match `v*` tags with GitHub Releases to deliver silent background downloads.

### Configuring the Repository (Owner & Repo)
To configure the application to look at your personal or organizational GitHub repository:
1. Open `package.json`.
2. Locate the `publish` block at the bottom.
3. Replace `<YOUR_GITHUB_USERNAME_OR_ORG>` with your GitHub username (e.g., `octocat`).
4. Replace `<YOUR_GITHUB_REPO_NAME>` with your repository name (e.g., `Chapter_One`).

Example:
```json
"publish": {
  "provider": "github",
  "owner": "octocat",
  "repo": "Chapter_One"
}
```

### Automatic CI/CD Pipeline
The `.github/workflows/release.yml` pipeline automatically builds and attaches your `.exe` to your GitHub Release page. 

**Requirements:**
1. You must give GitHub Actions write permissions to natively publish the releases (the workflow natively requests `permissions: contents: write`).
2. Your local repository must be synced cleanly to GitHub.

### How to Bump Version & Tag a Release
1. **Prepare Release**
   Run the helper script to mathematically bump your exact version globally:
   ```bash
   node scripts/set-version.js 1.2.5
   ```
2. **Commit the Bump**
   Commit your `package.json` and push it:
   ```bash
   git add package.json
   git commit -m "chore: bump version to 1.2.5"
   git push origin main
   ```
3. **Draft the Release via Git Tags**
   You natively assign a tag starting with `v` that identically matches the incremented version:
   ```bash
   git tag v1.2.5
   git push origin v1.2.5
   ```
   **That's it!** Pushing the tag automatically triggers `release.yml`. GitHub Actions will boot a Windows container, install dependencies, compile your `.exe`, generate the `latest.yml`, and securely host it on the "Releases" tab of your GitHub repository.

### How to Test Updates Locally (Mock Testing)
To truly verify your GitHub updater integrations locally *without* deploying to users:
1. **Push a Draft Release:** Tag `v1.2.5` but instead of pushing it directly, keep it as a *Draft* on GitHub so standard users don't see it (or push using a private test repo).
2. **Change your App token:** `electron-updater` allows you to define an environment variable `GH_TOKEN` securely to fetch Draft releases explicitly for dev QA testing.
3. **Compile Locally:** Because `electron-updater` explicitly isolates developer environments by intentionally throwing errors if you attempt to overwrite raw source code with `.exe` files (`app.isPackaged === false`), you cannot witness the actual installation execution through `npm start`.
   - Build local `v1.0.0`: `npm run build:win` -> Physically install it.
   - Boot the installed `1.0.0` app.
   - It will cleanly contact the GitHub endpoint, see `v1.2.5` tagged, trigger `download-progress`, and successfully restart & install.
